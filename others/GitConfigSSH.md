1. **生成了 SSH 密钥对（本地文件变更）**
   - 执行了 `ssh-keygen -t ed25519 -C "..."`。
   - **修改了什么**：在你的电脑 `C:\Users\25876\.ssh\` 目录下，**新增了两个文件**：`id_ed25519`（私钥，绝不能泄露）和 `id_ed25519.pub`（公钥，可以公开）。
   - **作用**：这相当于给你的电脑配了一把“专属数字钥匙”。
2. **把公钥注册到了 GitHub 服务器（远程账户变更）**
   - 你把 `id_ed25519.pub` 里的内容复制并粘贴到了 GitHub 网页后台（Settings → SSH keys）。
   - **修改了什么**：在你的 GitHub 账户后台，**新增了一条 SSH Key 记录**。
   - **作用**：告诉 GitHub 服务器：“这把公钥对应的电脑是我信任的设备，以后凭这把钥匙验证身份”。
3. **修改了本地仓库的远程地址（项目配置变更）**
   - 执行了 `git remote set-url origin git@github.com:ningqianlove/tech-notes.git`。
   - **修改了什么**：修改了你这个项目文件夹下隐藏的 `.git/config` 文件。把原来的 `https://github.com/...` 改成了 `git@github.com:...`。
   - **作用**：告诉 Git 客户端：“以后往这个仓库推代码，不要走 HTTPS 那个门（端口 443），改走 SSH 这个门（端口 22）”。

------

### 🤔 为什么这能根治你之前的“连接重置”报错？

- **之前的 HTTPS 协议（报错根源）**：走的是 **443 端口**，且需要依赖系统的代理设置和 SSL 证书验证。你的网络环境（可能是 VPN、代理软件或防火墙）在这一层发生了“数据劫持”或“证书篡改”，导致 Git 握手时直接收到“重置连接（Reset）”的指令，强行中断。
- **现在的 SSH 协议（成功路径）**：走的是 **22 端口**，且使用非对称加密（公钥/私钥）进行身份认证，**完全不经过**你 Windows 系统里设置的 HTTP 代理，也**不校验**那些被防火墙篡改的 SSL 证书。它走的是最底层的网络传输通道，只要你的网络能访问外网，它就能稳定传输。

### 📌 给个小贴士

你现在这个仓库已经彻底切到 SSH 了，以后所有的 `git push`、`git pull`、`git clone` 都不需要再输入账号密码，也**不会再因为代理问题而报 443 或 Reset 错误**。

如果以后新建别的仓库，记得在 `clone` 或 `remote add` 时，**复制地址时一定要选 SSH 标签下的地址**（即以 `git@github.com:` 开头的那个），而不是 HTTPS 的地址，就能永远避开今天遇到的这个坑了
